# Codex Ultra-Orchestrator v1 架构白皮书

## 1. 执行摘要

`Codex Ultra-Orchestrator v1` 是一套面向大型语言模型工程任务的、基于技能组合的轻量级智能体编排框架。它的目标，不是把 Codex 简单扩展成“能同时做更多事的单体 agent”，而是把它提升为一个具备需求归一化、决策完备规划、角色化分派、安全拦截、交叉审查、行为验证与可审计交付能力的多阶段工程协同中枢。

面对现代 coding agent 在复杂任务中常见的上下文崩溃、隐性回归、并发写冲突和结果不可追溯等问题，CUO v1 选择了一条相对克制但更可靠的路线：

> 在追求全自动化多智能体协同之前，先建立结构化、职责隔离、可回滚、可审计的编排基础。

因此，v1 并不从一个重型分布式调度平台起步，而是从协议、契约、技能包、最小执行账本与运行校验机制起步。它首先解决“工程纪律如何被机器执行”的问题，再逐步推进更高程度的自动化。

## 2. 架构溯源：三位一体的融合设计

CUO v1 不是对某一套现成方案的直接移植，而是融合了三类互补的方法论。

### 2.1 方法层：`superpowers`

`superpowers` 提供了技能化组织和流程复用的核心思路：

- 用高内聚技能替代臃肿单体 prompt
- 用可组合 workflow 替代一次性长上下文驱动
- 把优秀流程沉淀成标准化技能，而不是依赖临场发挥

它最大的价值，在于帮助系统回答“能力应该如何组织”。

### 2.2 控制平面层：`Ultra-Orchestrator`

`Ultra-Orchestrator` 提供了 CUO v1 的控制平面基础：

- 多阶段编排结构
- 风险审查门与输出审计
- 强制结构化产物
- 任务图思维
- orchestration log 与 vetter report

它回答的是“系统如何保持可控、可回溯、可审查”。

### 2.3 角色工程层：`gstack`

`gstack` 提供了工程团队式的角色工作流启发：

- 显式角色分工
- 上下游阶段交接
- sprint 式闭环
- `careful`、`freeze`、`guard` 这类实用护栏

它回答的是“一个工程化 agent 团队应该如何协同工作”。

### 2.4 三者结合后的架构含义

三者在 CUO v1 中形成明确分工：

- 用 `superpowers` 组织动作与技能树
- 用 `Ultra-Orchestrator` 保障控制、审计与风险门
- 用 `gstack` 约束角色流转与工程闭环

如果缺少其中任一层，系统都会出现明显短板：

- 只有技能组织，没有控制平面，系统容易松散
- 只有控制平面，没有角色流转，系统容易僵硬
- 只有角色流转，没有统一契约，系统容易失控

## 3. 核心设计原则

CUO v1 采用以下设计原则。

### 3.1 协议大于平台

先定义角色边界、状态流转、结构化契约与安全规则，再讨论底层自动化基础设施。

### 3.2 契约驱动而非记忆驱动

系统中不同阶段的协作，不建立在“保留整段历史对话”之上，而建立在明确的数据契约之上。任务状态、交接对象、风险结论和验收条件，都必须通过结构化对象表达。

### 3.3 上下文防火墙

下游角色默认不继承上游完整对话历史，只读取其完成当前阶段所必需的结构化产物与文件指针。这一原则用于降低 Token 污染、历史偏见继承与无关上下文膨胀。

### 3.4 基于产物的交接

交接必须通过明确产物完成，例如：

- `TaskManifest`
- `WorkPackage`
- `AgentResult`
- `ExecutionLedger`
- `RunOutput`

任何“仅存在于自然语言上下文中的约定”都不视为可靠交接。

### 3.5 集成前必审

未经规格对齐审查和工程质量审查的结果，在系统中视为未完成状态，拒绝直接集成。

### 3.6 图论与锁驱动的并发

并行不是抽象口号，而是机器可执行约束。只有同时满足以下条件，任务才允许并行派发：

- 在任务依赖图中已达到可执行状态
- 所声明的 `owned_paths` 互不相交
- 未被其他活跃任务持有写锁

### 3.7 审计即功能

系统必须不仅交付结果，还交付过程证据，明确记录：

- 做了什么
- 为什么这样做
- 哪些风险被识别过
- 哪些验证已经完成
- 哪些不确定性仍然存在

## 4. 状态机闭环与反馈工作流

CUO v1 不再使用脆弱的单向瀑布式流程，而采用带回滚与重试约束的有限状态机。

固定阶段为：

1. `Intake`
2. `Plan`
3. `Dispatch`
4. `Execute`
5. `Review`
6. `QA`
7. `Deliver`
8. `Retro`

### 4.1 Intake：需求摄入

职责：

- 将模糊请求归一化为结构化 intake artifact
- 提炼成功标准、约束、上下文与任务类型
- 判断是否需要前置风险探针

产物：

- intake artifact

### 4.2 Plan：决策完备规划

职责：

- 解决架构歧义
- 拆解任务 DAG
- 生成 `TaskManifest`
- 明确读写边界、依赖关系与非目标

产物：

- plan
- `TaskManifest`
- 初始 `WorkPackage`

### 4.3 Dispatch：受控派发

职责：

- 解析可执行任务
- 依据角色与边界下发 `WorkPackage`
- 更新 `ExecutionLedger`
- 检查并发条件与写锁冲突

产物：

- 派发后的 `WorkPackage`
- 更新后的 `ExecutionLedger`

### 4.4 Execute：受限执行

职责：

- Worker 在授权范围内完成工作
- 对瞬时失败执行有限重试
- 对硬阻塞立即上报

约束：

- 默认每个工作包设定最大重试次数
- 超出重试次数后必须升级处理，不允许无限循环

产物：

- `AgentResult`

### 4.5 Review：双重审查

职责：

- 规格审查：验证结果是否偏离计划目标
- 工程审查：验证质量、证据、防御性与回归风险

反馈链路：

- `Review -> Reject -> Execute`

如果审查失败，必须把具象化失败上下文带回执行阶段，而不是仅返回模糊的“重做”。

### 4.6 QA：行为验证

职责：

- 脱离代码实现细节，验证用户行为与关键场景
- 关注 happy path、error path 与 regression risk

反馈链路：

- `QA -> Failed -> Execute`
- 对严重架构偏差，允许 `QA -> Failed -> Plan`

这保证 QA 不只是“收尾确认”，而是真正具备拦截与回滚能力。

### 4.7 Deliver：交付封装

职责：

- 汇总用户可消费结果
- 聚合 `orchestration_log`
- 聚合 `vetter_report`
- 总结残留风险与下一步

### 4.8 Retro：过程复盘

职责：

- 提炼有效策略
- 记录失败模式
- 为后续编排策略优化提供依据

## 5. 角色纪律模型

角色纪律是系统最容易被忽视、却最重要的防御层之一。CUO v1 定义七个核心角色：

- `Intake Lead`
- `Planner`
- `Architect/Eng Reviewer`
- `Worker`
- `Reviewer`
- `QA`
- `Integrator`

### 5.1 各角色的禁止事项

`Intake Lead`

- 禁止凭空创造用户未提出的需求

`Planner`

- 专职进行问题建模与任务拆解
- 禁止直接进入大范围业务实现

`Worker`

- 仅在授权工作包范围内工作
- 禁止擅自跨越 `owned_paths`
- 禁止自行填补重大设计空白

`Reviewer`

- 在验收条件不明确时，必须拒绝放行

`QA`

- 必须包含回归思维
- 不仅验证新增路径，也验证旧链路是否被破坏

`Integrator`

- 负责汇总、拼接、整合和报告
- 禁止在该阶段偷偷重构已通过审查的他人结果

### 5.2 为什么角色纪律是系统级防御

在复杂任务中，失控往往不是因为模型不会写代码，而是因为不同阶段混在一起：

- planner 偷偷编码
- worker 偷偷做架构判断
- reviewer 在无标准下“感觉通过”
- integrator 偷偷重写别人产物

角色纪律的意义，在于把“职责漂移”显式转化为系统不允许的行为。

## 6. 共享数据契约

这些契约是系统的单一事实来源。

### 6.1 `TaskManifest`

表示任务蓝图，至少包含：

- `id`
- `goal`
- `context`
- `success_criteria`
- `constraints`
- `owned_paths`
- `dependencies`
- `expected_outputs`
- `risk_level`
- `non_goals`

### 6.2 `WorkPackage`

表示派发给具体角色的原子工作单元，至少包含：

- `task_id`
- `role`
- `scope`
- `owned_paths`
- `input_artifacts`
- `acceptance_checks`
- `report_format`

### 6.3 `AgentResult`

表示执行结果回传，至少包含：

- `task_id`
- `status`
- `summary`
- `changed_paths`
- `artifacts`
- `evidence`
- `risks`
- `followups`

### 6.4 `ExecutionLedger`

表示全局运行账本，至少记录：

- 当前阶段
- 各任务状态
- 依赖满足情况
- 重试计数
- blockers
- 待审查项
- 集成状态

### 6.5 `RunOutput`

表示最终交付态，顶层强制包含：

- `final_deliverable`
- `orchestration_log`
- `vetter_report`

## 7. 账本所有权与更新机制

这是 v1 现阶段必须明确的一条工程约束。

### 7.1 推荐机制：脚本主导账本更新

`ExecutionLedger` 不应由 LLM 在长上下文中直接输出整段大 JSON 并覆写。现阶段更稳妥的机制是：

- LLM 输出结构化意图或阶段产物
- 上层宿主脚本或状态机引擎解析这些产物
- 由脚本精确更新账本中的对应字段

这样做的原因很明确：

- LLM 不擅长在长会话中稳定维护大 JSON 的结构一致性
- 脚本更适合承担字段级更新、冲突检测与持久化职责
- 这也为后续引入轻量状态机引擎留下接口

### 7.2 v1 文档上的结论

因此，白皮书对账本的定义应当是：

- `ExecutionLedger` 是控制平面工件
- 其解释权和更新权优先属于宿主编排层
- LLM 负责生成阶段产物，而不是充当账本数据库本身

## 8. 并发模型：DAG 与写锁约束

“安全并行”在 v1 中必须被定义成可执行规则，而不是宽泛口号。

### 8.1 DAG 条件

一个任务只有在依赖满足时才进入可执行集合。对运行时来说，这等价于：

- 上游依赖已经完成或被标记为满足
- 当前任务没有未解决阻塞

### 8.2 写锁条件

一个任务只有在其 `owned_paths` 未与其他活跃任务发生写范围重叠时，才允许并行派发。

这相当于引入了路径级 mutex lock 概念：

- 同一资源只能由一个活跃写任务持有写锁
- 只读探索可并行
- 写任务与共享写任务必须串行

### 8.3 v1 的现实落地粒度

现阶段白皮书适合把这套约束写成“并发协议”，而不是提前承诺一个完整图调度器已经存在。也就是说：

- 原则现在就应写进文档
- 自动化调度引擎可以在后续阶段实现

## 9. 上下文经济：Artifact-Driven Handoff

“上下文防火墙”是非常适合当前阶段纳入白皮书的核心原则，而且这是 v1 可以立即受益的更新。

### 9.1 核心原则

下游角色默认不继承上游完整对话，只接收：

- 该阶段所需的结构化工件
- 最小必要文件指针
- 必要的失败上下文

### 9.2 为什么这很重要

没有上下文防火墙时，系统会出现三个问题：

1. Token 污染：无关历史不断堆积
2. 认知偏置继承：下游角色继承上游错误假设
3. 幻觉放大：账本越长，越容易出现不一致记忆

### 9.3 白皮书层面的写法

因此，文档应明确写出：

- Hand-off 建立在 artifact 上，不建立在 chat history 上
- Need-to-know context 是默认策略，而不是优化项
- 运行账本是状态索引，不是把所有自然语言对话永久塞进上下文

## 10. QA 的物理环境边界

这也是当前阶段应当明确写进白皮书的重要边界。

### 10.1 v1 的默认 QA 模式

在当前阶段，`qa-verify` 应被定义为“双模态 QA”：

- 基线模式：静态推演与基于现有测试结果的验证
- 增强模式：当宿主环境允许时，执行真实测试脚本或命令获取 stdout/stderr

### 10.2 v1 不应过度承诺的能力

当前白皮书不应把以下内容写成既成能力：

- 已内建无头浏览器验证框架
- 已具备通用端到端自动执行环境
- 已可在任意项目中自动生成并运行所有测试

### 10.3 更准确的定位

因此，v1 中 QA 的定位应是：

- 行为验证协议已经具备
- 动态执行能力取决于宿主环境与可用工具
- 浏览器级和复杂物理环境验证属于后续增强项

## 11. 双层安全控制平面

### 11.1 战略意图层：Risk Vetter

在调用外部系统、大范围写入或高影响操作前，必须先做风险定级：

- `LOW`：自动放行
- `MEDIUM`：放行，但附加 guardrails
- `HIGH`：挂起，等待更高权限批准
- `EXTREME`：默认硬阻断

### 11.2 战术执行层：Safety Guard

执行层包含三种实用模式：

- `careful`：执行高风险动作前进行提醒与确认
- `freeze`：限制文件系统写范围
- `guard`：同时开启 `careful` 与 `freeze`

其目标不是构造复杂安全平台，而是在当前阶段先把最实用的执行护栏落地。

## 12. 参考实现与试运行方式

CUO v1 当前已经以参考实现形式落地在本地技能目录中，包含：

- 主控技能：`ultra-orchestrator`
- 规划与编排技能：`clarify-and-intake`、`decision-complete-planner`、`dispatch-and-track`、`autoplan`
- 审查与交付技能：`spec-review`、`code-review`、`qa-verify`、`deliver-and-retro`
- 安全技能：`risk-vetter`、`safety-guard`
- 生命周期脚本：`new_run.py`、`validate_run.py`

### 12.1 试运行方式一：规划链路试驾

最轻量的试法，是直接使用 `$autoplan`，快速串起：

- Intake
- Planning
- Spec Review
- Refinement

这最适合验证：

- intake 是否稳定
- plan 是否 decision-complete
- spec review 是否能发现隐性空洞

### 12.2 试运行方式二：完整结构化 sprint

更完整的试法是：

1. 初始化 run 外壳
2. 由人类作为总控 orchestrator
3. 按阶段依次调用对应技能
4. 在阶段流转中不断校验产物是否合法

这适合验证：

- 状态流转是否清晰
- handoff artifacts 是否足够精简
- orchestration log 是否对复盘有价值

## 13. 当前阶段适合采纳的架构优化项

结合现阶段的实现成熟度，这些优化建议可以分成三类。

### 13.1 适合立即写入白皮书的

这些建议收益高、风险低，而且不会把尚未实现的能力写成既成事实：

1. 用“带反馈闭环的状态机”替代线性瀑布流
2. 引入上下文防火墙与 artifact-driven handoff
3. 把安全并行正式落成 DAG + 写锁约束
4. 删除元评论与外部 AI 评语，恢复白皮书的专业基调
5. 明确 ledger 由宿主脚本主导更新
6. 明确 QA 的物理环境边界

### 13.2 适合写成 v1 原则、但不夸大为已自动化实现的

这些内容适合进入文档，但应保持“协议已定义，自动化后续实现”的表达：

- 路径级 mutex lock
- 依赖图驱动的自动派发
- 更细粒度的失败上下文回传
- 阶段自动回滚与自动重唤醒

### 13.3 暂不适合写成当前能力的

这些方向是有价值的，但目前更适合写进 roadmap，而不是写成 v1 已具备：

- 完整的自动 event loop orchestrator
- 无需人工干预的守护进程式调度
- 通用浏览器级 QA 基础设施
- 全自动 policy engine

## 14. 当前刻意的架构边界

CUO v1 仍然保留以下刻意边界：

- 账本轻量化，优先验证协议而非数据库架构
- 调度轻量化，优先验证 handoff 质量而非全自动 event loop
- QA 轻量化，优先验证协议与验证路径，而非浏览器基础设施

这些边界并不是“系统不够先进”，而是 v1 对工程风险的主动控制。

## 15. 后续演进路线

### 15.1 第一优先级：真实 pilot

建议立即投入 2 到 3 个真实任务：

- 多文件功能开发
- 真实 bug 调查与修复
- 高风险但以 planning/review 为主的结构化任务

重点观察：

- Intake 是否稳定
- Plan 是否真正可执行
- Review 与 QA 是否具备打回能力
- Handoff artifacts 是否足够小而清晰

### 15.2 第二优先级：账本自动机

在 pilot 证明协议有效后，再建设轻量 Python 状态机引擎，用于：

- ledger 字段更新
- 上下游 agent 自动唤起
- payload 路由
- 状态变更约束

### 15.3 第三优先级：领域增强包

在通用内核之上增加：

- Frontend Delivery Pack
- Backend Migration Pack
- Debugging Pack
- Audit/Reporting Pack

## 16. 结语

CUO v1 证明了一个核心判断：

> 真正的智能，不只体现在单行代码生成速度上，更体现在组织复杂工程生成过程的纪律性上。

在生成式 AI 时代，经典软件工程纪律并没有失效。它只是需要被翻译成机器可读、机器可执行、机器可审计的结构化协议。

`Codex Ultra-Orchestrator v1` 的意义，正是在这里：它试图把“会做事的 agent”提升为“会组织复杂工作流的工程协作者”。
